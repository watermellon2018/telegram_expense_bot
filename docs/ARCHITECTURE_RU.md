# Документация архитектуры Telegram Expense Bot

## 1. Обзор проекта

### Что делает система
Система реализует Telegram-бота для учета личных и проектных финансов:
- добавление расходов и доходов;
- статистика за день/месяц/год;
- совместная работа в проектах с ролями;
- экспорт и PDF-отчеты;
- фоновые задачи (бюджетные уведомления, регулярные операции);
- метрики и структурированное логирование.

### Основные компоненты
- `main.py` — тонкая точка входа процесса.
- `app/*` — слой инициализации приложения и lifecycle.
- `handlers/*` — Telegram-слой (команды, сценарии диалога, callback-и).
- `utils/*` — инфраструктурные и прикладные утилиты (БД, форматирование, парсинг, визуализация, отчеты).
- `metrics.py` — Prometheus-метрики.

---

## 2. Архитектура

### Подробное описание модулей
- **Уровень представления**: `handlers/*`
  - принимает `Update/Context`;
  - валидирует вход пользователя;
  - вызывает функции доменной/инфраструктурной логики из `utils/*`;
  - формирует ответ Telegram.

- **Уровень инициализации приложения**: `app/*`
  - сборка `Application`;
  - подключение middleware и error handler;
  - lifecycle-хуки запуска/остановки;
  - регистрация фоновых задач.

- **Инфраструктурный уровень**: `utils/db.py`, `utils/logger.py`, `metrics.py`
  - подключение к PostgreSQL;
  - централизованное логирование;
  - метрики.

- **Прикладные доменные модули**: `utils/excel.py`, `utils/projects.py`, `utils/categories.py`, `utils/incomes.py`, `utils/report_generator.py`, `utils/visualization.py`
  - бизнес-операции учета и отчетности;
  - SQL-запросы и агрегации;
  - подготовка материалов для ответа пользователю.

### Поток данных
1. Пользователь отправляет команду/сообщение в Telegram.
2. PTB `Application` передает update в зарегистрированный handler.
3. Handler:
   - читает `context.user_data`;
   - при необходимости запрашивает/обновляет данные в БД через `utils/*`;
   - формирует текст/файл/график;
   - отправляет ответ пользователю.
4. Параллельно:
   - lifecycle планировщик запускает регулярные воркеры;
   - middleware и error handler пишут логи/метрики.

- `docs/`: проектная и архитектурная документация.
- `tests/`: модульные и регрессионные тесты.

## 4. Ключевые модули

- `main.py`: минимальная точка входа, запуск polling.
- `app/bootstrap.py`: сборка объекта `Application`, подключение handlers/middleware и обработчика ошибок.
- `app/lifecycle.py`: запуск/остановка, поднятие метрик, init/close DB pool, запуск scheduler.
- `handlers/registry.py`: централизованный и упорядоченный реестр handlers.
- `utils/helpers.py`: фасад совместимости (старые импорты продолжают работать).
- `utils/parsing.py`, `utils/formatting.py`, `utils/menu.py`, `utils/conversation.py`, `utils/context.py`: декомпозиция прежнего монолитного `utils/helpers.py`.
- `utils/db.py`: пул соединений и низкоуровневый SQL рантайм.
- `utils/excel.py`: ключевые операции по расходам (добавление/агрегация).
- `utils/projects.py`: проекты, роли, активный проект, приглашения.

## 5. Функции и методы (критично)

Ниже перечислены ключевые функции по важным модулям.

### Модуль: `main.py`

#### Функция: `main() -> None`
Описание:
Главная синхронная точка входа процесса бота.

Параметры:
- отсутствуют.

Возвращает:
- `None`.

Что делает:
1. Создает директорию данных.
2. Строит Telegram `Application` через `build_application`.
3. Запускает `run_polling`.

Побочные эффекты:
- запуск бесконечного polling loop;
- запись событий в лог.

Пограничные случаи:
- исключение в рантайм -> обработка в блоке `if __name__ == "__main__"` и логирование критической ошибки.

Использование:
- запускается при `python main.py`.

### Модуль: `app/bootstrap.py`

#### Функция: `build_application() -> telegram.ext.Application`
Описание:
Собирает и конфигурирует объект Telegram-приложения.

Параметры:
- отсутствуют.

Возвращает:
- готовый экземпляр `Application`.

Что делает:
1. Конфигурирует токен и lifecycle hooks (`post_init`, `post_stop`).
2. Регистрирует все handlers.
3. Подключает middleware и глобальный error handler.

Побочные эффекты:
- регистрация обработчиков и middleware в Application.

Пограничные случаи:
- ошибка токена/конфигурации приведет к падению при запуске polling.

Использование:
- вызывается из `main.main`.

### Модуль: `app/lifecycle.py`

#### Функция: `on_startup(application: Application) -> None`
Описание:
Хук запуска, вызываемый после инициализации PTB Application.

Параметры:
- `application: Application` — объект Telegram приложения.

Возвращает:
- `None`.

Что делает:
1. Создает директорию данных.
2. Инициализирует pool БД.
3. Поднимает Prometheus endpoint.
4. Инициализирует и запускает scheduler фоновых задач.

Побочные эффекты:
- внешние сетевые порты (метрики);
- соединения с БД;
- запуск фоновых джобов.

Пограничные случаи:
- недоступность БД или ошибка при старте scheduler прерывает запуск.

Использование:
- зарегистрирована в `Application.post_init`.

#### Функция: `on_shutdown(application: Application) -> None`
Описание:
Хук остановки для корректного завершения работы.

Параметры:
- `application: Application`.

Возвращает:
- `None`.

Что делает:
1. Останавливает scheduler.
2. Закрывает pool БД.

Побочные эффекты:
- остановка фоновых задач;
- закрытие открытых соединений.

Пограничные случаи:
- частичный shutdown (например, pool уже закрыт) логируется, но процесс может завершиться.

Использование:
- зарегистрирована в `Application.post_stop`.

### Модуль: `app/error_handlers.py`

#### Функция: `build_error_handler() -> Callable`
Описание:
Фабрика глобального обработчика исключений Telegram update конвейер.

Параметры:
- отсутствуют.

Возвращает:
- async-функцию `(update, context) -> None`.

Что делает:
1. Извлекает `user_id` и `update_id` при наличии.

Пограничные случаи:
- `update` может быть не `telegram.Update`; функция обрабатывает это безопасно.

Использование:
- добавляется в `application.add_error_handler(...)`.

Описание:
Создает глобальный пул `asyncpg` для всего процесса.

Параметры:
- отсутствуют (используются env-переменные).

#### Функция: `execute(query: str, *args, request_id: str | None=None) -> str`
Описание:
Выполняет SQL-команды изменения данных (`INSERT/UPDATE/DELETE`).

Параметры:
- `query`: SQL;
- `*args`: bind-параметры;
- `request_id`: ID корреляции.

Возвращает:
- строковый результат `asyncpg.execute`.

Побочные эффекты:

- логирование операций.

Пограничные случаи:
- SQL-ошибка -> исключение прокидывается выше.

Использование:

#### Функция: `fetch(query: str, *args, request_id: str | None=None) -> List[Record]`
Описание:
Выполняет `SELECT` и возвращает все строки.

Параметры/возврат/эффекты:
- аналогично `execute`, но только чтение.

Пограничные случаи:
- SQL-ошибка -> исключение.

Использование:
- read-операции по расходам/доходам/проектам.

Пограничные случаи:
- отсутствие данных по отдельным графикам не прерывает весь конвейер.

---

## 6. Точки входа

### Точки входа
- Процесс: `main.py -> main()`.
- Telegram рантайм:
  - команды (`/start`, `/add`, `/month`, ...);
  - текстовые сообщения (`handlers/expense.py::text_handler`);
  - callback query для inline-кнопок.
- Фоновые воркеры:
  - бюджетные уведомления;
  - регулярные расходы;
  - регулярные доходы.

### Основной конвейер выполнения
1. `main.main` создает `Application`.
2. `app.bootstrap.build_application` подключает handlers и lifecycle.
3. `on_startup` инициализирует DB/metrics/scheduler.
4. Update маршрутизируется в нужный handler.
5. Handler вызывает `utils/*` для чтения/записи.
6. Ответ отправляется пользователю.
7. На shutdown освобождаются ресурсы.

## 7. Расширение системы

### Куда добавлять новую функциональность
- Новая Telegram-команда: новый модуль в `handlers/` + регистрация в `handlers/registry.py`.
- Новый доменный use-case:
  - бизнес-логику в `utils/*` (или новый модуль);
  - handler только оркестрирует вызов и формат ответа.
- Новый общий helper:
  - парсинг -> `utils/parsing.py`;
  - формат текста -> `utils/formatting.py`;
  - клавиатуры/regex -> `utils/menu.py`;
  - контекст conversation -> `utils/context.py`/`utils/conversation.py`.

### Как не сломать архитектуру
- Не добавлять SQL в handlers.
- Не смешивать генерацию клавиатур и бизнес-агрегации в одном модуле.
- Сохранять порядок регистрации handlers в `handlers/registry.py`.
- При реорганизации helper-ов оставлять обратную совместимость через `utils/helpers.py`.
- Для критичных сценариев добавлять/обновлять тесты в `tests/`.

## 8. Технический долг / найденные проблемы

1. `utils/excel.py` исторически содержит и бизнес-логику, и SQL-детали, и legacy-совместимость.
   - Риск: высокая связность, сложность изменений.

2. Большие модули `utils/projects.py`, `utils/report_generator.py`, `handlers/expense.py`, `handlers/stats.py`.
   - Риск: трудно локально вносить изменения без регрессий.

3. Частично неоднородные контракты возврата (`None`, `False`, пустой dict).
   - Риск: скрытые ошибки и неоднозначная обработка в callers.

4. Сильная зависимость от рантайм-конфигурации окружения и внешних сервисов (Postgres, Telegram API, matplotlib-окружение).
   - Риск: трудности локального воспроизведения и CI.

5. В модуле `utils/helpers.py` долгое время была «свалка обязанностей» (исправлено фасадом + декомпозицией, но потребуются дальнейшие миграции прямых импортов на новые модули).

6. Тесты зависят от установленного `python-telegram-bot`; без зависимостей тестовый прогон недоступен.
   - Риск: локально можно пропустить регрессии, если окружение не поднято.
