# MVP кэшбэка: Техническая документация

## 1. Общий обзор системы

В боте реализован MVP-модуль кэшбэка, который считает **только потенциальный кэшбэк**.

Система не пытается определить фактическое банковское начисление.  
Расчёт выполняется по упрощённой модели:
- категория расхода;
- активные правила кэшбэка на месяц;
- выбранная карта (если указана) или «лучшее доступное правило» среди активных карт.

Ключевой принцип:
> Потенциальный кэшбэк рассчитан по категории расхода и может отличаться от фактического начисления банка.

---

## 2. Архитектура модуля кэшбэка

### Слои

1. **Хранилище данных (PostgreSQL)**
- `user_cards`
- `cashback_categories`
- `user_cashback_rules`

2. **Бизнес-логика**
- модуль: `utils/cashback.py`
- отвечает за CRUD сущностей и расчёт потенциального кэшбэка

3. **UI/UX Telegram**
- модуль: `handlers/cashback.py`
- отдельный раздел главного меню: `💳 Cashback`
- команды и экранные действия по картам, правилам, категориям и статистике

4. **Интеграция с отчётами/статистикой**
- `handlers/stats.py` (`/month`) — добавлен блок потенциального кэшбэка
- `handlers/report.py` — после PDF отправляется сводка по потенциальному кэшбэку за скользящие 12 месяцев

## 3. Описание сущностей

### 3.1 `user_cards`

Назначение: карты пользователя для правил кэшбэка.

Поля:
- `id` — PK
- `user_id` — FK -> `users.user_id`
- `card_name` — название карты
- `is_active` — признак активности (мягкое отключение)
- `created_at`
- `updated_at`

Ограничения:
- уникальность активного имени карты в рамках пользователя (регистронезависимо)
- карта не удаляется физически в MVP, а деактивируется

### 3.2 `cashback_categories`

Назначение: категории, по которым считается потенциальный кэшбэк.

Поля:
- `id` — PK
- `name`
- `is_global` — глобальная или пользовательская категория
- `user_id` — `NULL` для глобальных, `NOT NULL` для пользовательских
- `created_at`

Ограничения:
- глобальные категории уникальны по имени (регистронезависимо)
- пользовательские категории уникальны в рамках пользователя
- контроль области видимости через CHECK:
  - глобальная: `is_global = TRUE` и `user_id IS NULL`
  - пользовательская: `is_global = FALSE` и `user_id IS NOT NULL`

### 3.3 `user_cashback_rules`

Назначение: правила кэшбэка на конкретный месяц.

Поля:
- `id` — PK
- `user_id` — владелец правила
- `user_card_id` — FK -> `user_cards.id`
- `cashback_category_id` — FK -> `cashback_categories.id`
- `year`
- `month`
- `percent`
- `created_at`
- `updated_at`

Ограничения:
- `UNIQUE(user_card_id, cashback_category_id, year, month)`
- `percent` в диапазоне `[0, 100]`
- правила прошлых месяцев не перезаписываются автоматически

## 4. Подробное описание логики расчёта кэшбэка

## 4.1 Формула

Если выполнены условия:
1. есть правило на `year/month`;
2. категория расхода совпадает с cashback-категорией по имени (регистронезависимо, с нормализацией пробелов),

тогда:
`potential_cashback = amount * percent / 100`

иначе:
`potential_cashback = 0`

## 4.2 Правила выбора процента

### С выбранной картой
Берутся только правила этой карты (если карта активна).

### Без выбранной карты
Для каждой категории выбирается максимальный процент среди активных карт пользователя на месяц.

Это сделано как MVP-приближение, потому что транзакции не привязаны к карте.

## 4.3 Выходные агрегаты

На периоде (месяц / 12 месяцев) считаются:
- `total_spent`
- `potential_cashback`
- `effective_spent = total_spent - potential_cashback`
- `by_category` (потенциальный кэшбэк по категориям)

---

## 5. Ограничения системы (почему это не точный кэшбэк)

1. Нет связи конкретной транзакции с конкретной картой.
2. Нет MCC-анализа.
3. Нет учёта банковских правил округления и исключений.
4. Нет онлайн-интеграции с банком.
5. Сопоставление только по имени категории расхода.

Поэтому результат всегда маркируется как **потенциальный**.

---

## 6. Пограничные случаи

1. **Нет правил на месяц**
- кэшбэк = 0, effective_spent = total_spent.

2. **Категория расхода не совпала с cashback-категорией**
- по этой операции потенциальный кэшбэк = 0.

3. **Дубликат правила на месяц (карта+категория)**
- создание запрещается уникальным ограничением и проверкой в бизнес-логике.

4. **Деактивированная карта**
- правила такой карты игнорируются в расчёте.

5. **Пустой месяц (нет расходов)**
- все итоги = 0, дисклеймер всё равно выводится.

6. **Проектный режим**
- расчёт делается по расходам проекта при наличии прав просмотра.

---

## 7. Потенциальные улучшения (дорожная карта)

1. Привязка операции к карте при создании расхода.
2. Поддержка MCC и анти-правил (исключений).
3. Отдельные лимиты/кап на категорию и карту.
4. Расчёт «базовый + повышенный кэшбэк» с приоритетами.
5. Отдельная страница PDF-отчёта с аналитикой кэшбэка по месяцам/картам.
6. Рекомендации «какой картой выгоднее платить в категории».

## Детализация функций (вход/выход/логика/предположения)

Ниже перечислены ключевые функции и их контракты.

### `utils/cashback.py`

### `ensure_global_cashback_categories_exist() -> None`

Вход:
- нет.

Выход:
- `None`.

Логика:
- идемпотентно вставляет базовые глобальные категории.

Предположения:
- таблица `cashback_categories` уже создана миграцией.

### `list_user_cards(user_id: int, include_inactive: bool=False) -> List[Dict]`

Вход:
- `user_id`
- `include_inactive`

Выход:
- список карт пользователя.

Логика:
- выбирает карты по `user_id`, при необходимости включая неактивные.

Предположения:
- `user_id` существует или может не существовать (вернётся пустой список).

### `create_user_card(user_id: int, card_name: str) -> Dict`

Вход:
- `user_id`
- `card_name`

Выход:
- `{success: bool, card_id?, message?}`

Логика:
- нормализует имя;
- проверяет дубликат активной карты;
- создаёт карту.

Предположения:
- карта в MVP не удаляется физически.

### `edit_user_card(user_id: int, card_id: int, new_card_name: str) -> Dict`

Вход:
- `user_id`, `card_id`, `new_card_name`

Выход:
- результат обновления.

Логика:
- проверка существования карты у пользователя;
- проверка дубликата;
- обновление названия и `updated_at`.

### `deactivate_user_card(user_id: int, card_id: int) -> Dict`

Вход:
- `user_id`, `card_id`

Выход:
- результат деактивации.

Логика:
- `is_active = FALSE`, без hard delete.

### `list_cashback_categories(user_id: int) -> List[Dict]`

Вход:
- `user_id`

Выход:
- объединённый список глобальных и пользовательских категорий.

Логика:
- сначала гарантирует наличие глобальных категорий;
- затем читает глобальные + пользовательские.

### `create_custom_cashback_category(user_id: int, name: str) -> Dict`

Вход:
- `user_id`, `name`

Выход:
- результат создания категории.

Логика:
- валидация имени;
- проверка дубликата в пользовательской области;
- insert.

### `list_user_cashback_rules(user_id: int, year: int, month: int) -> List[Dict]`

Вход:
- `user_id`, `year`, `month`

Выход:
- список правил на период.

Логика:
- объединяет данные с картами и cashback-категориями для UI-отображения.

### `create_user_cashback_rule(...) -> Dict`

Вход:
- `user_id`, `user_card_id`, `cashback_category_id`, `year`, `month`, `percent`

Выход:
- результат создания правила.

Логика:
- проверка диапазона процента;
- проверка активной карты и доступности категории;
- проверка уникальности периода;
- insert.

Предположения:
- если запись на период уже есть, пользователь редактирует существующее правило.

### `edit_user_cashback_rule(user_id: int, rule_id: int, percent: float) -> Dict`

Вход:
- `user_id`, `rule_id`, `percent`

Выход:
- результат редактирования.

Логика:
- обновляет только процент и `updated_at`.

### `remove_user_cashback_rule(user_id: int, rule_id: int) -> Dict`

Вход:
- `user_id`, `rule_id`

Выход:
- результат удаления.

Логика:
- удаляет правило пользователя по id.

### `calculate_potential_cashback_for_period(...) -> Dict`

Вход:
- `user_id`, `year`, `month`, `project_id?`, `user_card_id?`

Выход:
- агрегированная сводка по потенциальному кэшбэку.

Логика:
1. получает эффективную карту правил на период;
2. читает расходы периода;
3. сопоставляет по нормализованному имени категории;
4. считает суммы и разбивку по категориям.

Предположения:
- сопоставление по имени категории достаточно для MVP.

### `calculate_potential_cashback_for_last_12_months(...) -> Dict`

Вход:
- `user_id`, `project_id?`, `user_card_id?`, `reference_date?`

Выход:
- агрегаты за скользящие 12 месяцев.

Логика:
- итерирует 12 месяцев назад и суммирует результат помесячного расчёта.

### `format_cashback_summary(summary: Dict, title: str | None=None) -> str`

Вход:
- `summary`, опциональный заголовок.

Выход:
- готовый текст для Telegram.

Логика:
- форматирует `total_spent`, `potential_cashback`, `effective_spent`, разбивку по категориям;
- всегда добавляет дисклеймер.

### `handlers/cashback.py`

### `cashback_menu(update, context) -> None`

Вход:
- Telegram update/context.

Выход:
- сообщение с меню раздела кэшбэка.

Логика:
- показывает отдельный раздел и дисклеймер.

### `cashback_cards(update, context) -> None`

Вход:
- update/context.

Выход:
- список карт + команды add/edit/deactivate.

### `cashback_rules(update, context) -> None`

Вход:
- update/context, опциональные аргументы `[month] [year]`.

Выход:
- список правил + команды управления.

### `cashback_categories(update, context) -> None`

Вход:
- update/context.

Выход:
- список глобальных и пользовательских категорий.

### `cashback_stats(update, context) -> None`

Вход:
- update/context (учитывается активный проект пользователя).

Выход:
- текстовая сводка потенциального кэшбэка за текущий месяц.

Логика:
- вызывает `calculate_potential_cashback_for_period`;
- выводит итог + разбивку по категориям + дисклеймер.
