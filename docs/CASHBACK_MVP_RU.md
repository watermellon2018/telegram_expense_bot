# MVP кэшбэка: Техническая документация

## 1. Общий обзор системы

В боте реализован MVP-модуль кэшбэка, который считает **только потенциальный кэшбэк**.

Система не пытается определить фактическое банковское начисление.  
Расчёт выполняется по упрощённой модели:
- категория расхода;
- активные правила кэшбэка на месяц;
- выбранная карта (если указана) или «лучшее доступное правило» среди активных карт.

Ключевой принцип:
> Потенциальный кэшбэк рассчитан по категории расхода и может отличаться от фактического начисления банка.

---

## 2. Архитектура модуля кэшбэка

### Слои

1. **Хранилище данных (PostgreSQL)**
- `user_cards`
- `cashback_categories`
- `user_cashback_rules`

2. **Бизнес-логика**
- модуль: `utils/cashback.py`
- отвечает за CRUD сущностей и расчёт потенциального кэшбэка

3. **UI/UX Telegram**
- модуль: `handlers/cashback.py`
- отдельный раздел главного меню: `💳 Cashback`
- команды и экранные действия по картам, правилам, категориям и статистике

4. **Интеграция с отчётами/статистикой**
- `handlers/stats.py` (`/month`) — добавлен блок потенциального кэшбэка
- `handlers/report.py` — после PDF отправляется сводка по потенциальному кэшбэку за скользящие 12 месяцев

## 3. Описание сущностей

### 3.1 `user_cards`

Назначение: карты пользователя для правил кэшбэка.

Поля:
- `id` — PK
- `user_id` — FK -> `users.user_id`
- `card_name` — название карты
- `is_active` — признак активности (мягкое отключение)
- `created_at`
- `updated_at`

Ограничения:
- уникальность активного имени карты в рамках пользователя (регистронезависимо)
- карта не удаляется физически в MVP, а деактивируется

### 3.2 `cashback_categories`

Назначение: категории, по которым считается потенциальный кэшбэк.

Поля:
- `id` — PK
- `name`
- `is_global` — глобальная или пользовательская категория
- `user_id` — `NULL` для глобальных, `NOT NULL` для пользовательских
- `created_at`

Ограничения:
- глобальные категории уникальны по имени (регистронезависимо)
- пользовательские категории уникальны в рамках пользователя
- контроль области видимости через CHECK:
  - глобальная: `is_global = TRUE` и `user_id IS NULL`
  - пользовательская: `is_global = FALSE` и `user_id IS NOT NULL`

### 3.3 `user_cashback_rules`

Назначение: правила кэшбэка на конкретный месяц.

Поля:
- `id` — PK
- `user_id` — владелец правила
- `user_card_id` — FK -> `user_cards.id`
- `cashback_category_id` — FK -> `cashback_categories.id`
- `year`
- `month`
- `percent`
- `created_at`
- `updated_at`

Ограничения:
- `UNIQUE(user_card_id, cashback_category_id, year, month)`
- `percent` в диапазоне `[0, 100]`
- правила прошлых месяцев не перезаписываются автоматически

## 4. Подробное описание логики расчёта кэшбэка

## 4.1 Формула

Если выполнены условия:
1. есть правило на `year/month`;
2. категория расхода совпадает с cashback-категорией по имени (регистронезависимо, с нормализацией пробелов),

тогда:
`potential_cashback = amount * percent / 100`

иначе:
`potential_cashback = 0`

## 4.2 Правила выбора процента

### С выбранной картой
Берутся только правила этой карты (если карта активна).

### Без выбранной карты
Для каждой категории выбирается максимальный процент среди активных карт пользователя на месяц.

Это сделано как MVP-приближение, потому что транзакции не привязаны к карте.

## 4.3 Выходные агрегаты

На периоде (месяц / 12 месяцев) считаются:
- `total_spent`
- `potential_cashback`
- `effective_spent = total_spent - potential_cashback`
- `by_category` (потенциальный кэшбэк по категориям)

---

## 5. Ограничения системы (почему это не точный кэшбэк)

1. Нет связи конкретной транзакции с конкретной картой.
2. Нет MCC-анализа.
3. Нет учёта банковских правил округления и исключений.
4. Нет онлайн-интеграции с банком.
5. Сопоставление только по имени категории расхода.

Поэтому результат всегда маркируется как **потенциальный**.

---

## 6. Пограничные случаи

1. **Нет правил на месяц**
- кэшбэк = 0, effective_spent = total_spent.

2. **Категория расхода не совпала с cashback-категорией**
- по этой операции потенциальный кэшбэк = 0.

3. **Дубликат правила на месяц (карта+категория)**
- создание запрещается уникальным ограничением и проверкой в бизнес-логике.

4. **Деактивированная карта**
- правила такой карты игнорируются в расчёте.

5. **Пустой месяц (нет расходов)**
- все итоги = 0, дисклеймер всё равно выводится.

6. **Проектный режим**
- расчёт делается по расходам проекта при наличии прав просмотра.

---

## 7. Потенциальные улучшения (дорожная карта)

1. Привязка операции к карте при создании расхода.
2. Поддержка MCC и анти-правил (исключений).
3. Отдельные лимиты/кап на категорию и карту.
4. Расчёт «базовый + повышенный кэшбэк» с приоритетами.
5. Отдельная страница PDF-отчёта с аналитикой кэшбэка по месяцам/картам.
6. Рекомендации «какой картой выгоднее платить в категории».

## 8. Snapshot аналитики по месяцам

### Что такое snapshot

`Snapshot` — это сохранённый итог расчёта кэшбэка за закрытый месяц.  
После создания snapshot месяц считается зафиксированным, и значения больше не пересчитываются.

### Зачем нужен snapshot

Без snapshot отчёты за прошлые месяцы считались бы «на лету» и могли бы изменяться при любых технических изменениях.  
Snapshot делает прошлые периоды стабильными и воспроизводимыми.

### Когда создаётся snapshot

Snapshot создаётся автоматически при первом запросе аналитики для **закрытого месяца**, если записи ещё нет.

Логика:
1. Пользователь запрашивает аналитику за месяц `M`.
2. Если месяц закрыт и snapshot уже существует: возвращается snapshot, без пересчёта.
3. Если месяц закрыт и snapshot отсутствует: выполняется расчёт, результат сохраняется в `cashback_monthly_snapshots`, затем возвращается сохранённый snapshot.
4. Если месяц не закрыт (текущий): расчёт всегда динамический.

### Как определяется закрытый месяц

Месяц закрыт, если он раньше текущего календарного месяца.

Примеры:
- Сегодня май 2026: апрель 2026 закрыт.
- Сегодня май 2026: май 2026 не закрыт.
- Любой месяц 2025 года закрыт.

### Почему данные не пересчитываются

Это бизнес-правило для immutable-аналитики:
- прошлые расходы не меняются;
- прошлые правила не меняются;
- отчёты за закрытые месяцы должны оставаться неизменными.

## 10. Функции snapshot-слоя (детально)

### `_is_closed_month(year: int, month: int, today: date | None = None) -> bool`

Что делает:
- определяет, закрыт ли целевой месяц относительно текущей даты.

Вход:
- `year`, `month`;
- `today` (опционально, для тестов/детерминированного вызова).

Выход:
- `True`, если месяц закрыт; иначе `False`.

Edge cases:
- некорректный месяц (`<1` или `>12`) возвращает `False`.

### `get_cashback_monthly_snapshot(user_id: int, year: int, month: int) -> Dict | None`

Что делает:
- читает snapshot за конкретный месяц.

Вход:
- `user_id`, `year`, `month`.

Выход:
- словарь summary (в формате, совместимом с форматтером), либо `None`.

Побочные эффекты:
- нет.

### `create_cashback_monthly_snapshot(user_id: int, year: int, month: int, summary: Dict) -> Dict`

Что делает:
- сохраняет snapshot с агрегатами месяца.
- если запись уже есть, не перезаписывает её (используется `ON CONFLICT DO NOTHING`), возвращает существующий snapshot.

Вход:
- `user_id`, `year`, `month`;
- `summary` (результат расчёта: суммы, число расходов, разбивка по категориям).

Выход:
- словарь summary из сохранённой записи + флаг `snapshot_created`.

Побочные эффекты:
- запись в таблицу `cashback_monthly_snapshots`.

### `_calculate_potential_cashback_for_period_dynamic(...) -> Dict`

Что делает:
- выполняет динамический расчёт без snapshot-логики.

Выход:
- агрегаты периода:
  - `total_spent`
  - `potential_cashback`
  - `effective_spent`
  - `expenses_count`
  - `by_category`

### `calculate_potential_cashback_for_period(...) -> Dict`

Что делает:
- публичная точка расчёта с автоматическим выбором между динамикой и snapshot.

Алгоритм:
1. Для проектного режима/выбора карты — динамический расчёт.
2. Для личной аналитики:
   - если месяц не закрыт — динамический расчёт;
   - если месяц закрыт:
     - вернуть существующий snapshot;
     - иначе рассчитать динамически и сохранить snapshot.

Гарантия:
- snapshot закрытого месяца не модифицируется после создания.

## 11. Интеграция в обработчики

- `handlers/cashback.py` (`/cashback_stats [month] [year]`) использует `calculate_potential_cashback_for_period`.
- `handlers/stats.py` (`/month`) использует тот же метод (для текущего месяца — динамика).
- `handlers/report.py` через `calculate_potential_cashback_for_last_12_months` агрегирует месяцы; закрытые месяцы берутся из snapshot автоматически через периодный расчёт.

---

## 12. Пример использования

### Команда

`/cashback_stats 4 2026`

Поведение:
1. Если сейчас май 2026 и snapshot за 04.2026 уже есть — возвращается сохранённый snapshot.
2. Если snapshot нет — расчёт выполняется один раз, сохраняется в `cashback_monthly_snapshots`, затем возвращается.
3. Повторный запрос за 04.2026 возвращает тот же snapshot без пересчёта.
